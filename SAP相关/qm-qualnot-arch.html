SAP QM · Quality Management · Business Object

质量通知单（Quality Notification）技术架构全景图

QM 不是一张简单表单，而是一个完整业务对象（Business Object），拥有 4 类创建入口、6 张核心透明表、状态驱动的数据模型。
4类创建入口
QM01人工创建事务码
6张核心透明表
JEST状态表驱动
🧭
核心认知：质量通知单在 SAP 里是业务对象而非表单——它由 问题 + 原因 + 措施 + 活动 + 状态 + 责任人 共同构成。理解它的 4 类创建入口、事务码与透明表，是从「会用 QM」走向「懂 QM 架构」的关键。
01 · 质量通知单的 4 类创建方式总览
1
人工直接创建
QM01
业务人员发现问题（生产异常 / 客诉 / 供应商问题）直接创建，最常用、最基础的一组。
用户发现质量问题
QM01 创建质量通知单
填写问题 → 分配责任 → 制定措施 → 关闭
2
检验批自动生成
QA11 / QE51N
QM 与检验管理结合，检验结果不合格时由系统触发生成通知单。
采购订单 101 收货
产生检验批 → QE51N 结果记录
QA11 使用决策「拒收」→ 生成通知单
3
从业务单据创建
CO11N / QM02
质量问题不是孤立的，可从生产 / 销售 / 采购 / 交货单钻出，体现 SAP「对象关联」设计思想。
生产订单 / 销售投诉 / 采购问题
CO11N 报工 / QM02 投诉
派生质量通知单 → 调查 → CAPA
4
接口 / API 创建
BAPI / IDoc
现代企业常见：MES 现场异常经接口调用 SAP，自动生成 QM 通知单。

02 · 方式 1：人工直接创建（QM01，最常用）
事务码作用类比 QM01创建质量通知单ME21N 创建采购订单 QM02修改质量通知单ME22N 修改采购订单 QM03显示质量通知单ME23N 查看采购订单
💡 创建示例：通知类型 Q1（质量问题）＋ 物料 100001 ＋ 批次 202607001 ＋ 描述「瓶盖破损」
03 · 方式 2：检验批自动生成
事务码作用关键动作 QA32修改检验批质量部门处理检验批 QE51N结果记录录入 PH / 外观等检验结果 QA11使用决策 ⭐接受 / 拒收 / 返工 / 报废，拒收可触发通知单
💡 场景：含量 98% 低于标准 99%~101% → 检验不合格 → 系统生成质量通知单

CO11N
从生产订单
生产报工时发现设备异常，可顺手创建质量问题，来源指向生产订单。
QM02
从销售投诉
客户投诉流程：客诉 → QM 通知单 → 调查 → CAPA，形成闭环。
QM01 · Q2

供应商问题用通知类型 Q2（供应商投诉），来源指向采购订单 / 供应商。
05 · 方式 4：接口 / API 创建（MES 集成）
MES → SAP 调用（BAPI_QUALNOT_CREATE）
CALL FUNCTION 'BAPI_QUALNOT_CREATE'
  EXPORTING
    NOTIF_TYPE = 'Q1'
    MATERIAL   = '100001'
    BATCH     = '202607001'
    DESCRIPTION = '设备污染'
  IMPORTING
    NOTIF_NUMBER = lv_qmnum.
MES 传参（JSON 示例）
{
  "material": "100001",
  "batch"   : "202607001",
  "description": "设备污染"
}

→ SAP 生成 QM 通知单
💡 技术形态：BAPI / IDoc / RFC / OData 均可，最常用 BAPI_QUALNOT_CREATE

07 · 质量通知单主要透明表（IT 经理视角）
透明表含义核心字段 QMEL质量通知单主表（核心）QMNUM / QMART / MATNR / CHARG / WERKS / QMTXT / ERNAM / ERDAT QMFE问题（Failure）QMNUM / FENUM / FEGRP / FECOD QMUR原因（Cause）原因代码，解释为何发生 QMSM措施（Task）MANUM / MATXT 整改措施 QMMA活动（Activity）通知供应商 / 召开质量会议等 JEST对象状态表 ⭐OBJNR / STAT（创建 / 处理中 / 完成 / 关闭）
💡 QMEL 示例：QMNUM 80000001 ｜ QMART Q1 ｜ MATNR A药品 ｜ CHARG 202607001
08 · 质量通知单与批次的关联
🔗
常见误区：以为 QMEL 直接关联批次。实际上 QMEL 虽有 MATNR / CHARG，但更完整的批次追溯依赖批次对象主数据：MCHA（批次主数据，工厂级）与 MCH1（跨工厂批次）。质量追溯要把 QMEL 与批次主数据结合看。

10 · 结合 MES 建设：SAP QM 最值得学习的三点
把 MES 质量异常模块从「记录系统」升级为「制造运营管理平台（MOM）」
① 一个异常不是一条记录，而是一个对象；② 问题、原因、措施分离（否则无法统计）；③ 状态驱动业务（而非「完成=Y」字段）。
建议直接参考 SAP 结构，设计如下 Z 表：
ZQM_NOTIFICATION
通知单主表
ZQM_FAILURE
问题
ZQM_CAUSE
原因
ZQM_ACTION
措施
ZQM_STATUS
状态流
闭环：创建 → 处理中 → 完成 → 关闭，与 SAP JEST 状态流一致，未来对接 QM 时天然对齐。
一句话总结
质量通知单 = 4 类入口（人工 / 检验批 / 业务单据 / 接口）＋ 6 张透明表（QMEL / QMFE / QMUR / QMSM / QMMA / JEST）＋ 状态驱动的业务对象。作为未来 IT 经理，你不需要背配置，但必须能回答：通知单怎么来、存在哪、状态怎么流、如何与 MES / 批次 / 订单关联。
