SAP 发出商品：SD 与 FI 的争论与和解

货已经给客户了，但能不能确认收入？SD 看物流、FI 看会计准则、仓库看实物——三个部门看的不是同一个问题
核心洞察：ERP 最难的地方不是单据，而是"业务事件和财务事件如何解耦"。PGI 解决物流真实性，发出商品解决价值确认延迟——两者不是冲突，是分层设计。

触发发出商品的典型业务场景

寄售 Consignment

药企最常见

药品发到医院/经销商库房，客户实际使用/销售后才结算。货物已离厂，但控制权未转移。

客户验收确认

设备/工程类

客户收货后需要质量验收、安装调试通过后才确认收入。未验收 = 不确认。

试用 / 委托代销

市场推广期

免费试用一段时间再决定是否购买，或委托第三方代销，售出才结算。

二SD 和 FI 的核心争论：PGI 后到底算不算销售完成？

SD 观点

"货已经给客户了，为什么不 PGI？"
  • 仓库已经发货，货不在企业手里
  • 客户已经收货、占有货物
  • 不 PGI 导致库存不准（系统有货、实物没了）
  • 影响 ATP / MRP / 盘点 / 销售绩效
关注：物流真实状态
VS

FI 观点

"不能这么早确认收入。"
  • 会计准则要求：控制权转移、风险转移、客户接受
  • 客户可能拒收、退货、未验收
  • 过早确认收入 → 审计风险、利润虚增
  • 税务时点也需要匹配
关注：价值确认合规
SD："不 PGI，库存不准。"   ← 矛盾 →   FI："PGI 后收入成本处理不符合财务要求。"
问题的本质：物流事件和财务事件时间点不同，需要解耦。

三SAP 的解决思路：物流与财务解耦

SAP 设计思想：PGI ≠ 一定收入确认。把物流事件和财务确认拆开为两个独立步骤。

核心方案：PGI + 发出商品科目（最常见）

① 创建销售订单 & 交货单
↓
② PGI 过账发货
↓
③ 物流：库存减少（实物已离开企业）
↓
④ 财务：借 发出商品 / 贷 库存商品
↑ 不是"主营业务成本"，而是"发出商品"
↓
⑤ 客户确认销售 / 验收 / 消耗
↓
⑥ 财务确认：借 主营业务成本 / 贷 发出商品
同时：借 应收账款 / 贷 主营业务收入
SD 获得什么？
物流动作可以先发生，库存实时减少，ATP 和 MRP 准确，交货状态可见。不用等财务确认就能反映真实库存。
FI 获得什么？
收入确认时机与会计准则对齐，发出商品仍算资产（不在库存但在财务报表中），月末无需人工调整。审计轨迹清晰。

三种方案横向对比：PGI 后库存与财务走哪条路？

方案做法优点缺点推荐
方案一
PGI → 发出商品		 仓库照常 PGI，财务凭证借 发出商品 / 贷 库存商品；客户签收后再借 主营业务成本 / 贷 发出商品。物流与财务时间点完全解耦。 权责发生制合规 · 库存实时准确 · 批次可追溯 · 审计轨迹完整 需管理发出商品余额 · 需客户签收确认机制 · 后续清理工作量 ✅
方案二
月底不 PGI		 实际发货但 SAP 暂不做 PGI，等客户签收后再统一 PGI + 确认成本。很多企业初期采用此方案。 操作简单 · 无需发出商品管理 系统库存虚高 · 订单状态不透明 · 批次追溯不准 · 仓库责任不清 · 月末大盘点大混乱 不推荐
方案三
在途库存		 设计多级库存地点：1000 工厂 → 2000 在途 → 3000 客户寄售，PGI 后物料转入在途库存地点。适用长运输周期、国际贸易场景。 实物位置可追踪 · 适合多段物流 需业务模式支持 · 库存地点管理复杂 · 仍需配合财务科目设计 特定场景

方案一是 SAP 标准推荐方案，也是本文重点讨论的路径。方案二看似简单但问题最多，不建议长期使用。

  • 货已经给客户了 → 系统应该反映

FI

FI 财务

  • 收入确认必须满足准则条件
  • 简单 PGI → 收入提前确认 → 审计风险
  • 月末大量调整 → 财务部门崩溃
WH

仓库/物流

看到的：实物在哪
  • 系统库存 1000，实物 800 → 每次盘点都打架
  • 不知道哪些货在客户那里
  • SAP 通过库存状态区分（非限制 / 发出商品 / 客户库存）
QM

QM 质量

看到：追溯责任
  • 药企：批次召回责任不因发货而消失
  • 必须知道：哪个批次 → 哪个客户 → 多少数量 → 什么时间
  • PGI + 发货凭证 = 完整批次追溯链

五最容易争论的三个业务点

争议 ① PGI 时间点

SD："装车就该 PGI" / FI："客户验收才算"
最佳实践：根据贸易模式定义。标准销售 = 装车 PGI；寄售/验收 = 装车 PGI 但财务进发出商品。

争议 ② 发出商品算库存吗？

SD："货没了，不算" / FI："资产还存在"
SAP 定位：不是普通库存，而是特殊库存/财务科目。财务报表算资产，但不出现在可用库存中。

争议 ③ 谁负责确认？

销售？仓库？财务？三方都觉得不是自己的事
必须定义 RACI，明确每个动作的责任人，避免月末互相甩锅。

六结合药企：最容易遇到的三个场景

场景 ① 商业配送

最常见

生产厂家 → 商业公司 → 医院。商业公司先收货，但医院实际销售后才结算。需要发出商品管理：货已到经销商，但收入确认延迟到终端消耗。

场景 ② 寄售 Consignment

SAP 标准功能

药品放到医院库房，医院使用后结算消耗。SAP 寄售流程：寄售补货 → 客户寄售库存 → 消耗过账 → 开票。全程不需要"发出商品"科目但逻辑一致。

场景 ③ 客户质量验收

高价值药品

客户收货后需质量确认或检验合格后才接受。未验收期间货物虽离开企业，但 GMP 批次追溯责任仍存在，且收入不能确认。

七架构视角：物流事件与财务事件如何解耦？

物流事件层（SD 视角）
PGI → 货离开企业 → 库存减少 → 批次记录
↓ 时间差
财务判断层（FI 视角）
判断：立即进成本？还是先进"发出商品"？
↓ 条件满足
收入确认层（准则要求）
控制权转移 + 风险转移 → 确认收入 & 成本
SAP 中的 PGI、GR（收货）、UD（质量放行）、TECO（技术完成）这些节点，本质都是在定义："什么时候系统承认某件业务事实发生？"SD 和 FI 的争论，实际是在争论企业经营规则，而不是 SAP 操作问题。
产品经理视角
每个关键业务事件必须有：状态转换（库存从"企业"到"发出"）+ 财务后果（进哪个科目）+ 责任归属（谁确认）。这三个维度缺一不可。
IT 架构视角
不要让 MES/WMS 替代 ERP 做库存确认和财务判断。MES 驱动物流动作，SAP PGI 作为最终状态确认点——层次清晰、责任明确、审计可追溯。

八月底场景：为什么这个问题在关账时特别突出？

一个具体场景：12 月末发货纠纷

12月28日 — 仓库发货
药品装车离开企业仓库，SAP 执行 PGI —— 库存减少
↓ 在途中
12月31日 — 财务关账
客户尚未收到货物。财务需要回答：这批货算什么？库存？成本？收入？
↓ 运输完成
1月2日 — 客户签收
客户确认收货，控制权正式转移。此时才满足收入确认条件。
如果 PGI 直接进主营业务成本
12 月报表：收入增加、成本增加 → 提前确认收入风险
但客户 12/31 尚未取得货物 → 审计风险、利润虚增
如果 PGI 进发出商品
12 月报表：库存→发出商品（仍是资产）
1 月客户签收后 → 结转成本 + 确认收入
→ 收入与成本在同一期间匹配，合规且可审计
月底关账时，财务必须回答：12 月 31 日企业的资产、收入、成本各是多少？如果跨月发货没有规范的发出商品管理，每月末都会出现大量人工调整。

各岗位日常操作变化

销售 / 销售内勤
以前 PGI 就结束。现在 PGI 后还需跟踪客户签收状态，催客户签收——否则财务不能确认收入。
仓库
PGI 操作本身几乎不变。但 PGI 不代表流程结束，后面还有"签收"环节。
财务
变化最大：月底不再只看库存，新增"发出商品余额核对"——哪些客户、哪些单据、为什么没签收、何时清理。
项目实施最大难点
签收数据从哪里来？不是 SAP 配置问题，而是业务规则问题。客户什么时候收到？依据什么？物流电子签收？客户 ERP？纸质回单？这决定了发出商品能否被清理。

十总结

SD 和 FI 争论的不是"要不要 PGI"，而是"PGI 之后走哪条财务路径"。SD 看物流——货离厂就该反映；FI 看准则——收入确认必须合规。SAP 的答案是分层解耦：PGI 照做（满足物流真实），但财务科目用"发出商品"（满足会计审慎）——不是冲突，是两层设计。

S/4HANA 特别提醒：你们已升级到 S/4HANA，建议项目启动前先确认：财务讨论的是 ECC 时代的"传统发出商品（自定义科目+增强）"，还是采用 S/4HANA 标准功能——如 Event-Based Revenue Recognition（基于事件的收入确认）或新的销售成本递延能力。S/4HANA 标准方案能减少大量 ABAP 自开发，各模块配置重点和事务码也会不同。
MES 记录生产事实，SAP 记录企业价值状态。
仓库"发货"是物流事实；客户"签收"是商业事实；财务"确认收入"是会计事实。
优秀 ERP 设计就是把这三个事实拆开，而不是强行绑定在一个操作上。
